## 0. 자동 적용 규칙 (Auto-apply) — 작성자/에이전트가 기계적으로 따를 것

새 페이지를 만들거나 사이트를 손볼 때, 네이버 관련해서는 **아래를 무조건 실행**한다. (각 항목 근거·상세는 본문 절 참조.)

1. **[색인 게이트]** 새/변경 페이지 발행 즉시 서치어드바이저 **`요청 → 웹페이지 수집`**에 canonical URL(후행 슬래시까지 정확히) 제출. sitemap·IndexNow를 1차 색인 경로로 쓰고 수동 수집요청은 핵심 URL 소수에만 배분(일일 한도·실행 규칙 = **§B-5 SSOT**). → §A-4, §B-5
2. **[IndexNow 우선]** 대량/자동화라면 네이버 **IndexNow(`요청 → 페이지 갱신 요청`, `/guide/indexnow-request`)** 또는 **크롤링 요청 API(`/guide/crawl-request-api`)** 로 발행 시점에 자동 핑. Bing/Yandex와 **동일 키** 공유. → §A-4.2~4.3, [`BING_YAHOO_INDEXNOW.md`](./BING_YAHOO_INDEXNOW.md)
3. **[robots]** 배포 전 `robots.txt`에 `User-agent: Yeti` 허용 + `Sitemap:` 라인 존재 확인. Yeti를 `Disallow`하거나 Cloudflare/Vercel WAF·봇룰이 Yeti를 차단하면 **네이버에서 통째로 사라진다** → Yeti는 예외처리. UA 하드매칭 금지, 부분문자열 `Yeti`만 매칭. → §A-5, **복붙 CF WAF 룰 §E-2**
4. **[소유확인 유지]** `<meta name="naver-site-verification">`를 **초기 HTML(SSR/정적)** 에 렌더. CSR-only 주입 금지. HTML 태그 방식은 **1년마다 재인증** 필요하니 태그 제거 금지. → §A-1.2
5. **[OG 필수·SSR]** 모든 페이지 `<head>`에 고유 `<title>` / `meta description` / `og:title` / `og:description` / `og:image` / `og:url` / `og:type` 를 **서버사이드로** 렌더. 네이버 스니펫·미리보기·카드는 **OG/title/description에 의존**한다(JSON-LD 리치결과 보장 없음). → §D-1
6. **[사이트맵]** `sitemap.xml`은 sitemaps.org 표준(파일당 5만 URL·압축전 50MB) 준수, 초과 시 `<sitemapindex>` 분할·gzip 허용. `<loc>` 도메인 = 서치어드바이저 등록 도메인(www/비www·http/https)과 **문자 그대로 일치**. RSS 병행 제출. → §A-2, §A-3
7. **[색인 배제]** noindex가 필요하면 robots.txt 크롤 차단이 아니라 `<meta name="robots" content="noindex">`(크롤은 허용)로 처리. 크롤이 막히면 noindex를 못 읽는다. → §A-5.3
8. **[AI 브리핑 인용성]** 정보 페이지는 **질문형 소제목 + 2~3문장 즉답 요약**으로 시작, 자격/절차/요금/서류는 **표·번호 리스트**로 구조화. 실제 네이버 질의("EPIK 자격 조건", "김치모바일 eSIM 개통 방법")와 소제목을 **1:1** 매칭. 발행 후 인용 여부는 **수동 샘플링으로 측정**(콘솔 지표 없음). → §B-8, §C-2
9. **[경험 신호 D.I.A+]** 정보 나열이 아니라 **날짜·구체 수치·1인칭 경험·자체 촬영 이미지**를 넣어 문서 경험성 강화. 스톡/AI 이미지만으로 채우지 않는다. → §B-2
10. **[출처 이원화]** 외부 홈페이지는 블로그/카페/인플루언서/지식iN 상위 블록에 **원천적으로 못 든다**. 네이버 상단이 목표면 **별도 네이버 블로그/인플루언서 채널**을 병행 운영하고, 사이트는 **'웹사이트' 블록 + AI 브리핑 인용**을 노린다. → §B-3, §B-4, §C-2
11. **[한국어 우선 색인]** 한국인 타깃 한국어 페이지는 반드시 사이트맵 포함 + 수집요청 체크. 네이버는 **한국어 페이지 위주로 색인·노출**한다. → §E-1
12. **[YMYL 강화]** korvia(비자/취업)·kimchi(통신/금융)는 원본·근거·최신성·작성자/날짜 표기를 강화해 네이버 웹문서 품질 기준(중복·자동생성 지양)을 충족. → §E-3
13. **[Cue: 언급 금지]** 2026-04-09 부로 Cue:·CLOVA X 종료. "Cue: 최적화" 같은 옛 지침 사용 금지 → **AI 브리핑 / AI 탭** 최적화로 표기. → §B-8
14. **[검증 후 기재]** 절차·한도·메뉴 라벨 수치는 발행 전 콘솔 실화면에서 재확인. `⚠` 표시 항목은 비공식/시점의존 값.

# PART A — 서치어드바이저 색인/제출 운영

## A-0. 한눈에 보는 운영 플로우

| 단계 | 메뉴 경로(웹마스터 도구) | 목적 | 소요 |
|---|---|---|---|
| 1. 사이트 등록 | 사이트 입력창 → `https://도메인/` 입력 | 관리 대상 추가 | 즉시 |
| 2. 소유확인 | (등록 직후) HTML 파일 업로드 **또는** HTML 태그(메타태그) | 소유권 증명 | 수분 |
| 3. 사이트맵 제출 | **요청 → 사이트맵 제출** | 수집 대상 URL 목록 전달 | 즉시 접수, 반영 수일 |
| 4. RSS 제출 | **요청 → RSS 제출** | 신규/갱신 콘텐츠 빠른 통지 | 즉시 접수 |
| 5. 개별 색인 요청 | **요청 → 웹페이지 수집** | 특정 URL 우선 수집 | 하루 한도 내 |
| 6. IndexNow/API 핑 | **요청 → 페이지 갱신 요청** (IndexNow) / 크롤링 요청 API | 발행 시점 자동 통지 | 즉시 |
| 7. 크롤 제어 | 서버 `robots.txt` + **검증 → robots.txt** | Yeti 허용/차단 검증 | - |
| 8. 진단·리포트 | **검증 → 웹페이지 최적화** / **리포트 → 사이트 최적화** | HTML 마크업·SEO 점검 | - |

> **핵심 원칙**: 소유확인만으로는 색인되지 않는다. **사이트맵/RSS 제출 + robots 허용(Yeti) + 콘텐츠 품질**이 함께 있어야 네이버 통합검색 '웹사이트/웹문서' 영역에 노출된다.

⚠ **메뉴 라벨 주의**: `요청 / 검증 / 리포트`의 상위 그룹 라벨·배치는 서치어드바이저가 주기적으로 개편한다. 하위 항목(`사이트맵 제출` / `RSS 제출` / `웹페이지 수집` / `robots.txt` / `웹페이지 최적화`)은 다수 출처로 확인되나, 발행 전 콘솔 실화면에서 최종 확인 권장.

## A-1. 사이트 등록 & 소유확인

### 1.1 등록
1. `searchadvisor.naver.com` 접속 → **네이버 아이디 로그인**(실명 계정 필요).
2. 우측 상단 **[웹마스터 도구]**.
3. 사이트 입력창에 **대표 주소 1개**를 프로토콜 포함 전체 URL로 입력(예: `https://www.korvia.com/`).
   - `http` vs `https`, `www` 유무는 **서로 다른 사이트로 취급**. 실제 서비스되는 canonical 주소 하나로 등록하고 나머지는 301로 통일.

### 1.2 소유확인 — 2가지 방식
공식: `help.naver.com/service/30010/contents/17594`

**방식 A. HTML 파일 업로드** (파일 접근 가능한 서버/정적호스팅 권장)
1. 소유확인 화면 → **[HTML 파일 업로드]** → 제공 파일(예: `naver1a2b3c….html`) 다운로드.
2. **도메인 루트 폴더**에 업로드.
   - **Next.js/Vercel**: `public/`에 넣으면 루트로 서빙(`https://도메인/naver….html`).
   - **WordPress**: 루트(`public_html`).
3. `https://도메인/naver….html`이 200으로 열리는지 확인 후 **[소유확인]**.

**방식 B. HTML 태그(메타태그)** (소스 수정 가능·루트 파일 업로드 곤란 시 권장)
1. **[HTML 태그]** → 메타태그 복사:
   ```html
   <meta name="naver-site-verification" content="고유해시값" />
   ```
2. 홈(최소) `<head>`에 삽입.
   - **Next.js (App Router)**: `app/layout.tsx`
     ```ts
     export const metadata = {
       verification: { other: { 'naver-site-verification': '고유해시값' } },
     };
     ```
   - **WordPress**: SEO 플러그인(Yoast/RankMath) 또는 헤더 훅.
3. 배포 후 실제 페이지 소스에 태그가 렌더되는지 확인 → **[소유확인]**.

> ⚠ **HTML 태그 방식은 1년마다 재인증** 필요. 태그를 지우지 말 것. 만료 알림 시 재확인.

### 1.3 소유확인 실패 흔한 원인
- 확인 파일이 **루트가 아닌 하위 폴더**에 업로드됨(경로 불일치).
- 파일명/해시 오타, 인코딩 문제.
- 서버 권한/방화벽/WAF로 외부(Yeti 포함) 파일 접근 불가(403).
- 메타태그가 **CSR로만 삽입**되어 초기 HTML에 없음 → SSR/정적 HTML에 포함해야 안전.
- 캐시/CDN 반영 지연.
- 등록 도메인(`www` 유무)과 실제 서빙 도메인 불일치.

## A-2. 사이트맵(sitemap.xml) 제출
공식: `help.naver.com/service/30010/contents/17629`

### 2.1 절차
1. `sitemap.xml`을 **루트**에 업로드: `https://도메인/sitemap.xml`.
2. 웹마스터 도구 → 대상 사이트 → **요청 → 사이트맵 제출**.
3. **경로만** 입력(도메인 뒤, 예: `sitemap.xml`) → **[확인]**.

### 2.2 형식·한도 (sitemaps.org 표준)
| 항목 | 값 |
|---|---|
| 파일당 최대 URL 수 | **50,000개** |
| 파일 최대 용량(압축 전) | **50MB (52,428,800 bytes)** |
| 압축 | **gzip(.gz) 지원** |
| 초과 시 | **사이트맵 인덱스**(`<sitemapindex>`)로 분할 제출 |
| 인코딩 | UTF-8, `<loc>`는 절대 URL |

> ⚠ 5만/50MB/gzip은 **sitemaps.org 국제 표준값**이며 네이버가 표준을 따른다고 안내한다. "네이버가 자체 명시한 상한 숫자"는 공식 도움말에서 1:1로 확정 확인되지 않음 — 표준 기준으로 설계하면 안전. (과거 표준이 10MB였다가 50MB로 상향된 이력 있음.)

### 2.3 실무 팁
- 반응형이면 **단일 사이트맵**으로 충분(PC/모바일 분리 불필요).
- `<lastmod>`를 정확히(거짓 lastmod 남발 금지 — 신뢰 저하).
- `<loc>` 도메인 = 등록 도메인과 **문자 일치**(www/비www, http/https).
- 제출 후 "성공/처리중" 상태 표시. **색인은 즉시가 아님(수일~수주).**
- Next.js는 `app/sitemap.ts`(MetadataRoute.Sitemap)로 자동 생성. 상세 템플릿·priority 규칙 → [`ROBOTS_SITEMAP_RSS_TEMPLATES.md`](./ROBOTS_SITEMAP_RSS_TEMPLATES.md).

## A-3. RSS 제출
공식: 고객센터 'RSS 및 사이트맵 제출 가이드'

### 3.1 절차
1. RSS 피드 주소 확인(WordPress `https://도메인/feed/`, `/rss`).
2. 웹마스터 도구 → **요청 → RSS 제출** → 피드 경로 입력 → **[확인]**.

### 3.2 특징·주의
- RSS는 **갱신 잦은 사이트(뉴스/블로그)** 에 유리 — 신규 글을 Yeti 방문 전에 알려 빠른 반영 유도.
- **RSS 본문 `<link>` 주소 = 등록 사이트 주소** 여야 함. 대표 도메인을 `mydomain.com`으로 쓰는데 서치어드바이저에 `www.mydomain.com`을 등록하면 *"등록한 사이트의 주소와 RSS 본문의 link 주소가 다릅니다"* 오류.
- ⚠ 피드가 담는 **최근 항목 수가 과다하면 제출 반려** 사례 보고됨(항목 수 줄여 승인 후 원복하면 이후 수집 정상). 정확한 상한(예: 100)은 공식 미명시 — flag.
- **사이트맵(전체 구조) + RSS(신규/갱신 신호) 병행 제출** 권장.

## A-4. 개별 URL 색인 요청 · IndexNow · 크롤링 요청 API

### 4.1 요청 → 웹페이지 수집 (수동 개별 요청)
1. 웹마스터 도구 → **요청 → 웹페이지 수집**.
2. 색인시킬 URL을 **전체 절대주소**로 입력 → **[확인]**.
3. "수집 요청 완료" 표시. 이후 Yeti가 방문·수집·색인 판단.

**일일 한도 및 함정**
- **일일 쿼터 존재**(아래 크롤링 요청 API가 초과 시 `errorCode 4021005 "하루 요청량 초과"`를 반환 = 확정). 구체 수치·헤징·실행 규칙은 **§B-5 SSOT 참조**(값 중복 방지 — 여기서 숫자 재기입 안 함).
- **후행 슬래시(trailing slash)**: 사이트 canonical 형태와 정확히 일치(`.../글이름/`). 슬래시 누락 시 네이버가 "리디렉션된 페이지"로 보고 수집 반려 가능.
- **리디렉션 함정**: `http→https`, `비www→www`, trailing-slash 불일치 등 301이 걸린 URL은 수집 제한/실패. Next.js `trailingSlash` 정책·WordPress 정규화와 **제출 URL을 반드시 일치**시킬 것.
- 수집 요청은 **색인을 보장하지 않음**(품질/중복/robots 차단 시 제외).
- 대량 신규는 개별 요청보다 **사이트맵/RSS/IndexNow**가 정공법. 개별 요청은 "긴급/핵심 랜딩페이지"에 아껴 쓴다.

### 4.2 요청 → 페이지 갱신 요청 (IndexNow) 🆕
공식: `searchadvisor.naver.com/guide/indexnow-request`

- **네이버 서치어드바이저는 IndexNow 프로토콜을 지원한다.** 콘솔 **`요청 → 페이지 갱신 요청`** 에서 키 발급·확인 후, 페이지 생성/수정/삭제 시 URL을 즉시 통지(핑)할 수 있다.
- IndexNow는 **Bing·Yandex·Naver 등이 공유하는 개방형 프로토콜** — **키 파일 하나로 여러 엔진에 동시 통지**. 즉 KORVIA는 IndexNow 키를 한 번 심으면 Bing과 네이버를 함께 커버한다. → 상세 구현·키 파일 배치는 [`BING_YAHOO_INDEXNOW.md`](./BING_YAHOO_INDEXNOW.md).
- 응답 상태코드(공식 가이드 표기): `200`(정상), `403`(key 무효), `422`(URL이 key 정보와 불일치), `429`(과도한 요청), `500`(서버 에러).
- **권장**: 발행 파이프라인(Vercel 빌드 후 훅, n8n, WordPress 저장 훅)에서 **자동 핑**. 수동 '웹페이지 수집'은 IndexNow가 커버 못 하는 케이스·급한 단건에만.

> ⚠ IndexNow의 네이버 지원은 공식 가이드 URL(`/guide/indexnow-request`)로 확인되나, 콘솔 UI의 정확한 키 발급 위치·라벨은 개편 가능 — 최초 설정 시 실화면 확인.

### 4.3 크롤링 요청 API 🆕
공식: `searchadvisor.naver.com/guide/crawl-request-api`

- 프로그램적으로 URL 수집/삭제를 요청하는 **REST API**. 자동화(대량 발행·삭제 동기화)에 사용.
- 제약(공식 표기):
  - **payload 최대 2MB**.
  - `update`와 `delete`에 동일 URL이 있으면 **delete 우선**.
  - 하위 URL 중 **제휴(등록·소유확인)된 사이트만** 처리.
  - 초과 시 `errorCode 4021005` = "하루 요청량을 초과하였습니다" / `5001030`·`5001031` = 서버 에러.
  - 성공 시 `errorCode 0 / message "Success"`.
- KORVIA 활용: n8n 워크플로우에서 신규/변경/삭제 URL을 배치로 update/delete. → 자동화 패턴은 [`ROBOTS_SITEMAP_RSS_TEMPLATES.md`](./ROBOTS_SITEMAP_RSS_TEMPLATES.md) 및 n8n 하네스와 연동.

## A-5. robots.txt & Yeti 크롤러 제어
공식: `searchadvisor.naver.com/guide/seo-basic-firewall`(검색로봇 확인), `.../seo-basic-create`(robots.txt 예시)

### 5.1 Yeti User-Agent (공식)
일반 웹문서 수집:
```
Mozilla/5.0 (compatible; Yeti/1.1; +https://naver.me/spd)
```
JS/CSS 등 **리소스 수집 시** 브라우저 UA 뒤에 Yeti가 붙는 형태:
```
Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36 (KHTML, like Gecko; compatible; Yeti/1.1; +https://naver.me/spd) Chrome/W.X.Y.Z Safari/537.36
```
- 봇 안내 URL: `https://naver.me/spd`.
- ⚠ **세부 버전(1.1·Chrome 빌드)과 프로토콜은 예고 없이 변경**. 현행 공식 firewall 페이지는 `https://naver.me/spd`를 쓰나 구가이드/미러는 `http://`로도 표기 → **전체 UA 하드매칭 금지, 부분문자열 `Yeti`만 매칭**.
- **역DNS 검증**(스푸핑 방지): 방화벽/WAF 화이트리스트 실무에서는 UA만 믿지 말고 요청 IP를 역조회해 **도메인이 `.naver.com`으로 끝나는지** 확인:
  ```bash
  # 접속 IP 역DNS 조회 → *.naver.com 이어야 진짜 Yeti
  host 203.0.113.10        # → x.x.x.x.naver.com 형태면 정상
  # 정방향 재확인(선택): host <위 도메인> 이 원래 IP로 돌아오는지
  ```
- Cloudflare/Vercel 등에서 Yeti를 **차단하지 않도록 허용**(외국인 대상 사이트는 해외 IP 봇 차단 룰에 Yeti가 걸릴 위험 — §E-2).

### 5.2 robots.txt 규칙 (네이버 가이드 예시)
- 위치: **루트** `https://도메인/robots.txt` (반드시 최상위).
- 전체 허용:
  ```
  User-agent: *
  Allow: /
  ```
- Yeti 명시 허용 + 사이트맵(권장 최소형):
  ```
  User-agent: Yeti
  Allow: /
  Disallow: /admin/
  Disallow: /api/

  User-agent: *
  Allow: /

  Sitemap: https://www.example.com/sitemap.xml
  ```
- ⚠ 전체 `Disallow: /`이면 어떤 페이지도 노출되지 않음. **관리자/검색결과/개인정보 등 비색인 대상 패턴만** Disallow.
- **검증 → robots.txt** 에서 규칙 파싱/차단 여부를 콘솔에서 테스트.
- 전체 템플릿(AI 크롤러 포함) → [`ROBOTS_SITEMAP_RSS_TEMPLATES.md`](./ROBOTS_SITEMAP_RSS_TEMPLATES.md).

### 5.3 페이지 단위 색인 제어
- 색인 배제는 **robots.txt 크롤 차단이 아니라** `<meta name="robots" content="noindex">`(크롤 허용). 크롤이 막히면 noindex 메타를 못 읽어 오히려 잔존 색인이 남는다.

# PART B — 노출·랭킹 신호 (C-Rank · DIA · 노출영역 · AI 검색)

> ⚠ **정확성 경계**: 네이버는 랭킹 가중치를 공식 수치로 공개하지 않는다. 아래 "40/30/20/10" 같은 C-Rank 구성비, DIA vs DIA+ 내부 구조는 **업계 리버스엔지니어링 추정**이며 공식 값이 아니다. 절차·한도·크롤러명 등 검증 가능한 항목만 단정한다.

## B-0. 큰 그림 — 네이버는 "문서 SEO"가 아니라 "출처 + 영역" 싸움

구글이 **URL(페이지) 단위**로 품질을 매기는 것과 달리, 네이버 통합검색은 두 층위다.

1. **어떤 영역(블록)에 들어갈 자격이 있는가** — 블로그/카페/인플루언서/지식iN/플레이스/웹사이트/뉴스/이미지/동영상/쇼핑 + 이를 감싸는 **스마트블록**. 영역별 진입 자격(출처 유형)이 다름.
2. **그 영역 안에서 몇 위인가** — 여기서 C-Rank(출처)·DIA+(문서)가 작동.

→ **함의**: 외부 홈페이지(Next.js/WordPress)는 블로그/카페/인플루언서/지식iN/플레이스 영역에 **원천적으로 못 들어간다.** 외부 사이트가 노릴 수 있는 건 ① **'웹사이트/사이트' 영역**, ② **뉴스(제휴 시)**, ③ **AI 브리핑/AI 탭 인용 소스**, ④ 쇼핑/플레이스(계정 등록형). → KORVIA류는 **네이버 블로그·인플루언서 채널을 별도 운영**하면서 외부 사이트는 웹사이트 영역 + AEO로 접근하는 **이원 전략**이 정석. (§B-3~B-4, §C-2)

## B-1. C-Rank — 출처(블로그/계정) 신뢰도·인기도

**정의(네이버 공식 블로그)**: 문서 자체가 아니라 **문서의 출처(블로그)**가 "얼마나 오래 믿을 수 있는 정보를 특정 주제로 꾸준히 생산했는가"를 본다. 2015년 적용, 이후 "검색 주제별 사용자 패턴"까지 반영해 고도화.

| 축 | 의미 | 실무 신호 |
|---|---|---|
| **Context**(맥락) | 주제 집중도·전문성 | 한 블로그가 한 주제(예: 'EPIK/비자')만 깊게 파는가 |
| **Content**(콘텐츠) | 정보 품질 | 원본성, 정보 밀도, 품질의 연속성 |
| **Chain**(연쇄) | 소비/생산 반응 | 검색 유입·스크랩·체류 등 실제 사용 패턴 |

→ 이 셋으로 **Creator(출처)의 신뢰도+인기도** 산정.

⚠ **업계가 흔히 인용하는 구성비(비공식 추정)**: 주제 전문성 40 / 활동 지속성 30 / 사용자 반응 20 / 콘텐츠 품질 10. **정확한 값 아님** — "전문성 > 지속성 > 반응"의 **상대 우선순위**로만 사용.

**실무 규칙**
- 계정(블로그)당 **하나의 좁은 주제**로 누적. 잡블로그는 C-Rank가 약하다.
- 신규 계정은 C-Rank 자본 0 → **초기 상위노출 거의 불가(유리천장)**. 수개월 주제 일관 포스팅 필요.
- 경쟁도 높은 정보성 질의(YMYL 유사 영역: 의료·금융·법률·취업)에서 특히 강하게 작동.

## B-2. D.I.A / D.I.A+ — 개별 "문서"의 경험·정보성

- **D.I.A (Deep Intent Analysis)**: 2018년~. 문서에 **실제 경험/정보가 담겼는지**를 평가(체류·클릭·문서 구성 등). C-Rank 약한 신규 블로그도 "문서가 진짜 좋으면" 노출되게 하는 **문서 단위 구제 로직**.
- **D.I.A+**: 질의를 더 잘게 분석. (추정 구성) 질의 의도 분석기 + 문서 패턴 분석기 + 문서 확장 모듈 + 피드백 반영. "이 질의는 후기/방법/비교/정의 중 무엇을 원하는가"를 판별해 의도에 맞는 문서를 끌어올림.

**실무 규칙**
- **경험 신호를 문서 안에 명시**: 직접 찍은 사진, 날짜, 구체 수치, 1인칭 경험("직접 EPIK 지원해보니…").
- 질의 의도별 포맷: '방법'→단계형, '후기'→경험+사진, '비교'→표, '정의/자격'→구조화 요약.
- C-Rank(출처) × D.I.A+(문서)는 **AND**에 가깝다 — 좋은 출처 + 좋은 문서가 겹칠 때 최상위.
- ⚠ 2026 업계 관측: AI 브리핑·"Agent N" 재편 이후 **"독창적 경험 증명"** 신호가 더 강조된다는 보도가 있으나 공식 수치는 없음 — 경험 신호 강화 방향은 안전.

## B-3. 통합검색 영역별 진입조건 (2024 재편 반영)

⚠ **2024년 VIEW 탭 종료** → 인기글이 **스마트블록**으로 흡수, **블로그 탭 / 카페 탭**으로 분리(개인 생산=블로그, 커뮤니티=카페).

| 영역 | 진입 주체 | 핵심 진입/상위 신호 | 외부 홈페이지 가능? |
|---|---|---|---|
| 블로그 탭 | 네이버 블로그(외부 일부 수집) | C-Rank + DIA+, 원본성·경험 | ❌ 네이버 블로그 계정 필요 |
| 카페 탭 | 네이버 카페 글 | 카페 활성도·질의적합·경험 | ❌ |
| 스마트블록 | 위 출처를 질의의도별 재배열 | 세분 의도 매칭, 구조화·요약성 | 간접(블로그로) |
| 인플루언서 | **승인된 인플루언서 계정**만 | 키워드챌린지 등록, 주제전문성, 팬·반응 | ❌ |
| 지식iN | 아무나 답변 | 질문 적합·명확·신뢰답변 | ❌ 개인 답변 |
| 플레이스 | **스마트플레이스 등록 업체** | §B-6(유입 자연성·행동전환) | 계정 등록형만 |
| **웹사이트/사이트** | **모든 외부 도메인** | 서치어드바이저 등록·수집·품질 | ✅ **외부 사이트의 정문** |
| 뉴스 | 언론 제휴사 | 언론 제휴 필요 | ❌ 제휴 아니면 |
| 이미지/동영상 | 전 출처 | alt/파일명/원본성/영상 메타 | ✅ 부분 |

→ **KORVIA 4개 사이트의 1순위 영역 = '웹사이트/사이트' + AI 브리핑 인용.** 블로그·인플루언서 노출을 원하면 **별도 네이버 채널**을 운영해야 한다.

## B-4. 네이버 블로그 vs 외부 홈페이지 — 노출 격차와 외부 상위 전략

- **격차 원인**: 상위 블록(블로그/카페/인플루언서/지식iN/플레이스)은 **네이버 내부 UGC 전용**. 외부 도메인은 태생적으로 불가. 통합검색 상단은 대부분 네이버 자체 콘텐츠 → 외부 사이트는 아래 '웹사이트' 블록으로 밀림.
- **외부 사이트가 그나마 상위로 가는 길** (상세 §C-2):
  1. **서치어드바이저 등록 + 수집 보장** — 색인 안 되면 순위 자체가 없다.
  2. **구조화·명확성** — 제목·H태그·요약문·FAQ·정의문 명확 → 스마트블록/AI 브리핑이 인용하기 쉬운 형태.

# PART D — 네이버의 구조화 데이터/메타 처리 현실

## D-1. title / description / OG (확정적으로 중요)
- 네이버는 스니펫·미리보기·카드 생성에 `<title>`, meta `description`, **Open Graph**를 적극 활용.
- **필수**: 페이지마다 고유 `<title>`, `description`, `og:title`, `og:description`, `og:image`, `og:url`, `og:type`. (OG는 미리보기/카드에 특히 중요.)

## D-2. 구조화 데이터(JSON-LD) — ⚠ 현실
- 네이버 가이드는 구조화 데이터를 인정하고 페이지 유형을 명확히 하는 마크업을 권장하며, **Open Graph + Microdata/JSON-LD** 혼용을 언급한다.
- schema.org 표준 타입(`Article`/`BlogPosting`, `Organization`, `WebSite`, `BreadcrumbList`, `Product`, `LocalBusiness` 등)은 넣는 게 일반적이며 **구글과 공용**.
- ⚠ **네이버가 "리치결과로 실제 렌더링/노출 보장"하는 스키마 타입의 공식 확정 목록은 단일 공식 페이지로 확인되지 않는다.** 구글과 달리 네이버의 리치결과 지원 범위는 제한적일 수 있음.
- **안전 전략**: **schema.org JSON-LD를 넣되(구글용으로도 유효), 네이버 노출 신뢰는 OG/title/description에 둔다.** (JSON-LD 카탈로그·빌더 → [`SCHEMA_JSONLD_CATALOG.md`](./SCHEMA_JSONLD_CATALOG.md).)
- ⚠ **구글 caveat(참고)**: 구글은 2026년 기준 FAQ/HowTo 리치결과를 사실상 폐지했다. JSON-LD FAQ/HowTo를 넣어도 구글 리치결과는 안 뜬다 — 그래도 **AI 인용·의미 구조화 목적**으로는 유효하니 유지. 상세 → [`GOOGLE_SERP_STRUCTURED_AI_2026.md`](./GOOGLE_SERP_STRUCTURED_AI_2026.md).

# PART E — 한/영 이중언어 · 외국인 대상 · YMYL

## E-1. 언어·색인 원칙
- 네이버는 **한국어 페이지 위주로 색인·노출**. 한국인 타깃 한국어 URL이 **사이트맵에 확실히 포함**되도록.
- 이중언어 사이트에서 **영어 페이지는 네이버 오가닉 기여가 작다** — 외국인 타깃 영어 콘텐츠의 검색 유입은 주로 **구글(영어)** 이 담당([`GOOGLE_SEARCH_CONSOLE_INDEXING.md`](./GOOGLE_SEARCH_CONSOLE_INDEXING.md), [`AEO_GEO_PLAYBOOK_2026.md`](./AEO_GEO_PLAYBOOK_2026.md)).
- ⚠ AI 브리핑의 영어/일본어 등 다국어 인용 범위는 유동적 — 외국인 대상(korvia/kimchi 영문)의 네이버 AI 브리핑 인용 기회는 발행 시점 재확인.

## E-2. 방화벽/WAF 함정 (외국인 사이트 특유)
- 외국인 대상 사이트는 해외 IP 봇을 광범위 차단하는 WAF 룰을 두기 쉬운데, 이때 **Yeti가 함께 차단되면 네이버 색인이 통째로 죽는다.** Cloudflare/Vercel 봇룰에 **Yeti(부분문자열) + `.naver.com` 역DNS 예외**를 명시(§A-5.1). (korvia·passome·act 3개 사이트가 전부 Cloudflare 뒤이므로 아래 룰을 그대로 적용.)
- **복붙용 Cloudflare WAF custom rule** — 액션 **Skip**(또는 Allow), 검색봇을 봇룰보다 먼저 통과시킨다(Daum의 Daumoa도 함께 예외 → [`DAUM_KAKAO_SEARCH.md`](./DAUM_KAKAO_SEARCH.md)):
  ```
  # Expression (Skip: Super Bot Fight Mode·Managed Rules를 건너뜀)
  (http.user_agent contains "Yeti") or (http.user_agent contains "Daumoa") or (http.user_agent contains "Daum")
  ```
  - ⚠ **룰 실행 순서**: 이 커스텀 Skip/Allow 룰이 **Super Bot Fight Mode·Managed Rules보다 먼저** 평가되도록 배치. Super Bot Fight Mode는 UA allowlist를 무시하고 차단할 수 있어 **'허용했다고 믿었는데 여전히 차단'되는 사일런트 실패**가 흔하다 → SBFM의 **'Verified Bots 허용' 토글**을 켜거나, 위 Skip 룰로 봇룰 자체를 건너뛴다.
  - ⚠ UA 예외만으로는 스푸핑에 취약 → **`.naver.com` / `.daum.net` 역DNS 검증**을 병행(§A-5.1).
  - **배포 후 검증**: 서치어드바이저 **검증 → robots.txt** 결과보기로 Yeti 통과 확인, Daum은 **webmaster.daum.net '로봇룰'** 결과보기로 확인.

## E-3. YMYL (비자/취업 = korvia, 통신/금융 = kimchi)
- 네이버 웹문서 품질 기준(중복·자동생성 지양)을 특히 엄격히 충족: **원본성 · 근거(출처 링크·수치) · 최신성(작성/수정일) · 작성자 신뢰 신호.**
- 자격/절차/요금 등 **사실 정보는 표·리스트로 정확히** — AI 브리핑 인용성과 신뢰 둘 다 확보.
- 규제·최신 정책은 날짜 명시 + 정기 갱신(비자 요건·통신 요금은 변동 잦음).
